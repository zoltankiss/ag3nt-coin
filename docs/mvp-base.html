ag3nt MVP — Base + Smart Contracts

Trustless settlement for agent work, end-to-end in one focused day.

Proposed Base Sepolia 2026-05-24 · Pivot proposal — supersedes the Cosmos appchain direction in SPEC.html

TL;DR

  • Goal: Tixswarm pays Hermes workers (running on your Mac minis) in ag3nt to generate ticket titles via Claude — fully on-chain settlement, end-to-end in one day.
  • Why this works economically: Tixswarm currently pays Anthropic per call for title generation. Hermes workers run on flat-rate Claude Pro subscriptions — marginal cost ≈ $0. You own both sides; you capture the arbitrage as a closed-loop demo.
  • Where it runs: Base Sepolia — a free testnet that is a real Ethereum-compatible chain (L2). Same code, same wallets, same tools as mainnet; no real money at risk.
  • What we build: (1) an Ag3nt ERC-20 token contract with a hard cap of 21M, (2) a JobEscrow contract that holds payment until work is delivered, (3) the hermes Node daemon that watches the chain and does the work, (4) the Tixswarm code that posts jobs.
  • What we explicitly defer: jury / disputes, EAS vouches, PageRank reputation, on-chain mining, smart-contract wallets, mainnet. Each has a clear future home in the roadmap.

The Premise

This whole MVP exists to demonstrate one specific dynamic: agents can pay other agents to do LLM work, with the chain providing trustless settlement. Until that loop runs end-to-end with real on-chain value moving, everything else (reputation, jury, mining curves) is theory.

The smallest setup that proves the dynamic is one you already own:

RoleIdentityWhat it does
Agent A (buyer)Tixswarm backendHas long descriptions, wants short titles. Holds ag3nt, posts paid jobs.
Agent B (worker)hermes on macmini64Has a Claude Pro subscription (marginal cost ~$0). Watches the chain, claims jobs, returns titles, earns ag3nt.
Agent B2 (competitor)hermes on macmini24Same as above, different wallet. Adds the "two competing workers" dynamic that turns a payment demo into a market demo.

Why this beats a "real" multi-party demo

Both sides being your own infrastructure means: no external dependencies, no recruiting workers, no token distribution problem, no real-world UX to debug. The arbitrage is real (you're literally paying Anthropic today for work you could do with idle Pro capacity), so the demo isn't a toy — it's a small but genuine cost-reduction loop.

Crypto Primer

If you've ignored crypto for a few years, here's a 5-minute orientation. Every concept here will appear later in the spec.

1. Chains, L1 vs L2

A blockchain is a public database that lots of independent computers maintain copies of. Every change requires every computer to agree. This is slow and expensive, but it means no single party controls the state — useful when two parties don't trust each other.

Ethereum is the most popular general-purpose blockchain — it runs programs (smart contracts), not just balance transfers. Ethereum's runtime is called the EVM (Ethereum Virtual Machine). Programs are written in Solidity and compiled to EVM bytecode.

Ethereum mainnet ("L1") is expensive — each transaction costs $1–$50 depending on congestion. L2s (layer 2s) like Base, Arbitrum, and Optimism are separate chains that batch up many transactions, post a single proof to Ethereum, and inherit Ethereum's security. Transactions on Base cost roughly $0.01–$0.10 — 100–1000× cheaper. From a developer's perspective, Base is just "Ethereum with cheap fees" — same Solidity, same wallets, same tools.

Testnet = a parallel copy of the chain, with no real money, used for development. Base Sepolia is Base's testnet. You get free testnet ETH from a faucet. Your contracts deploy and run identically — only the network ID and the dollar value of the tokens change.

Why Base specifically

Three reasons: (1) cheap gas, so transaction costs don't drown the demo; (2) standard EVM, so we get the entire Ethereum ecosystem of tools, audited contracts, and infrastructure; (3) Coinbase backs it, so it has staying power. The same MVP would work on Arbitrum or Optimism with no code changes.

2. Tokens vs Gas — the most important distinction

Common confusion: ETH vs ag3nt

These are two different things and you need both:

  • ETH = the fuel for the chain itself. Every transaction (sending a token, calling a contract, anything) costs a tiny amount of ETH as gas. Think of it as postage on a letter. You need ETH in your wallet to do anything on the chain. On testnet, ETH is free from a faucet. On mainnet, you buy it.
  • ag3nt = the currency for agent work. Our brand new token, defined by our contract. It's worth what we say it's worth (initially nothing, eventually whatever the market decides). It's used to pay for jobs.

Concretely: Tixswarm pays a Hermes worker 10 ag3nt for a title. The transaction itself costs Tixswarm ~0.0001 ETH in gas. Both have to be in Tixswarm's wallet.

A token in Ethereum-land is a smart contract that maintains a balance map: address → uint256. The ERC-20 standard is a small set of functions every well-behaved token contract has (balanceOf, transfer, approve, etc.). Wallets, exchanges, and other contracts all know how to interact with any ERC-20 because the interface is identical. Our ag3nt token will be an ERC-20.

"Hard cap" means the contract's logic prevents the total supply from ever exceeding a fixed number — in our case, 21,000,000 ag3nt, following Bitcoin's precedent. The cap is enforced by the contract's _mint function reverting if a mint would push total supply above the cap. Once the contract is deployed and the owner key is removed, the cap is mathematically guaranteed.

3. Smart Contracts

A smart contract is a program deployed to the EVM. Once deployed, its code is public, its state is public, and anyone can call its functions (subject to whatever access rules the contract itself enforces). You can think of it as a tiny stateful HTTP server that lives on the blockchain — but with three big differences:

  • Every function call costs gas. The caller pays. Cost depends on how much computation and storage the call uses.
  • State changes are persistent and public. Every state variable is readable by anyone. Privacy is opt-in (via off-chain commitments).
  • The code is usually immutable. Once deployed, you can't change a contract's bytecode (unless you built upgradability in, which has trade-offs). For our MVP, both contracts are immutable after deploy.

A contract has read functions (free, called view or pure) and write functions (cost gas, change state). It emits events — log entries that are cheap to write and easy for off-chain code to listen to.

4. Wallets & Keys

An address (e.g. 0x742d35...) is the public identifier of an account. Anyone can send ag3nt or ETH to your address. Spending from your address requires the private key — a 256-bit secret. The private key signs transactions; the chain verifies the signature against the address.

There are two kinds of accounts:

  • EOA (Externally Owned Account) — controlled directly by a private key. Simple. What we'll use for the MVP.
  • Contract Account — code, no private key. Acts when called. ERC-4337 smart wallets (Week 2+) are a hybrid: a contract account that uses a private key to authorize itself, with extras like spending limits, paymasters (so you can pay gas in ag3nt instead of ETH), and session keys. Powerful but more complex.

Key handling

Private keys live in .env files, never in git. Each .env is generated on the target machine and never transits. For mainnet, we'd move to hardware-backed keys (Lambda's KMS, hardware wallets, or AWS Nitro) — but for Sepolia, .env is fine because there's no money at risk.

5. Events & Listening

When a contract function fires, it can emit events. Events are searchable log entries — cheap to write, easy to query. Off-chain programs subscribe to event streams via RPC and react. This is how Hermes will know a new job is available: it subscribes to the JobPosted event on the escrow contract.

// Solidity side
event JobPosted(uint256 indexed jobId, address indexed buyer, uint256 payment, bytes32 inputHash);

// TypeScript side (viem)
escrow.watchEvent.JobPosted({}, {
  onLogs: (logs) => logs.forEach(handleNewJob)
});

Events are how the "real world" hooks onto the chain. Without them, off-chain workers would need to poll every block — slow and expensive.

Architecture

Smart Contracts

Two contracts, both deployed once to Base Sepolia, both immutable after deploy. Total Solidity ≈ 120 lines.

Ag3nt.sol — the token Day 1

A standard ERC-20 with a 21M hard cap and a single privileged minter (the escrow contract). The cap is enforced inside mintForJob; once total supply reaches it, no further mints succeed.

// SPDX-License-Identifier: MIT
pragma solidity ^0.8.24;

import "@openzeppelin/contracts/token/ERC20/ERC20.sol";

contract Ag3nt is ERC20 {
    uint256 public constant MAX_SUPPLY = 21_000_000 ether;  // 21M * 10^18
    address public immutable marketplace;

    constructor(address _marketplace, address initialHolder, uint256 preMint)
        ERC20("ag3nt", "AGNT")
    {
        require(preMint <= MAX_SUPPLY, "pre-mint exceeds cap");
        marketplace = _marketplace;
        _mint(initialHolder, preMint);   // seed Tixswarm with operating balance
    }

    /// @notice Called by the JobEscrow when a job settles successfully.
    function mintForJob(address worker, uint256 reward) external {
        require(msg.sender == marketplace, "only marketplace");
        require(totalSupply() + reward <= MAX_SUPPLY, "cap");
        _mint(worker, reward);
    }
}

Why is mint authority on the escrow, not the deployer?

Two reasons. (1) Credible neutrality: from day one, nobody — including you — can mint outside the rules. (2) Forward compatibility: when we add the Bitcoin-style halving curve, it lives inside the escrow's mintForJob call, with no upgrade needed on the token contract itself.

JobEscrow.sol — the marketplace Day 1

The state machine for a single job: Posted → Claimed → Submitted → Released (or Disputed). The contract holds the buyer's ag3nt in custody from postJob until release, so neither side can rug the other mid-trade.

contract JobEscrow {
    enum State { None, Posted, Claimed, Submitted, Released, Disputed }

    struct Job {
        address buyer;       // Tixswarm
        address worker;      // Hermes (set on claim)
        uint96  payment;     // ag3nt the worker receives on release
        uint96  workerBond;  // ag3nt the worker stakes when claiming
        uint40  submittedAt; // timestamp of result submission
        State   state;
        bytes32 inputHash;   // commitment to the prompt
        bytes32 resultHash;  // commitment to the result (set on submit)
        bytes   result;      // cleartext title (revealed on submit)
    }

    uint256 public constant DISPUTE_WINDOW = 60 seconds;
    Ag3nt   public immutable token;
    uint256 public nextJobId;
    mapping(uint256 => Job) public jobs;

    event JobPosted   (uint256 indexed id, address buyer, uint96 payment, bytes32 inputHash);
    event JobClaimed  (uint256 indexed id, address worker);
    event JobSubmitted(uint256 indexed id, bytes32 resultHash, bytes result);
    event JobReleased (uint256 indexed id, address worker, uint96 payment);
    event JobDisputed (uint256 indexed id);

    function postJob(uint96 payment, bytes32 inputHash) external returns (uint256 id) {
        token.transferFrom(msg.sender, address(this), payment);
        id = ++nextJobId;
        jobs[id] = Job(msg.sender, address(0), payment, 0, 0,
                       State.Posted, inputHash, 0, "");
        emit JobPosted(id, msg.sender, payment, inputHash);
    }

    function claimJob(uint256 id, uint96 bond) external {
        Job storage j = jobs[id];
        require(j.state == State.Posted, "not open");
        token.transferFrom(msg.sender, address(this), bond);
        j.worker = msg.sender;
        j.workerBond = bond;
        j.state = State.Claimed;
        emit JobClaimed(id, msg.sender);
    }

    function submitResult(uint256 id, bytes32 resultHash, bytes calldata result) external {
        Job storage j = jobs[id];
        require(j.state == State.Claimed && msg.sender == j.worker, "not yours");
        j.resultHash   = resultHash;
        j.result       = result;
        j.submittedAt  = uint40(block.timestamp);
        j.state        = State.Submitted;
        emit JobSubmitted(id, resultHash, result);
    }

    function release(uint256 id) external {
        Job storage j = jobs[id];
        require(j.state == State.Submitted, "not submitted");
        require(block.timestamp >= j.submittedAt + DISPUTE_WINDOW
                || msg.sender == j.buyer, "wait or be buyer");
        j.state = State.Released;
        token.transfer(j.worker, j.payment + j.workerBond);
        emit JobReleased(id, j.worker, j.payment);
    }

    function dispute(uint256 id) external {
        Job storage j = jobs[id];
        require(j.state == State.Submitted && msg.sender == j.buyer, "no");
        require(block.timestamp < j.submittedAt + DISPUTE_WINDOW, "too late");
        j.state = State.Disputed;
        // Day-1 resolution: buyer refunded, worker's bond slashed back to buyer.
        token.transfer(j.buyer, j.payment + j.workerBond);
        emit JobDisputed(id);
    }
}

What "optimistic settlement" means in practice

The buyer doesn't have to do anything for the worker to get paid — payment auto-releases 60 seconds after submission. The buyer only intervenes if something's wrong. This makes the happy path zero-effort (which it should be: 99% of jobs will be fine) and makes the unhappy path explicit (the buyer must affirmatively click "dispute"). On day one, the dispute is the buyer's unilateral call. Future versions (MVP-2) replace this with a stake-weighted jury so the buyer can't grief the worker.

Hermes Worker Day 1

A Node.js daemon, one instance per Mac mini. It subscribes to JobPosted events, claims jobs it can do, executes them by spawning claude --headless, and submits the result on-chain.

// hermes/src/index.ts
import { createPublicClient, createWalletClient, http, webSocket } from "viem";
import { privateKeyToAccount } from "viem/accounts";
import { baseSepolia } from "viem/chains";
import { spawn } from "node:child_process";
import { keccak256, toBytes } from "viem";

const account = privateKeyToAccount(process.env.HERMES_KEY as `0x${string}`);
const wallet  = createWalletClient({ account, chain: baseSepolia, transport: http() });
const reader  = createPublicClient   ({ chain: baseSepolia, transport: webSocket() });
const escrow  = { address: process.env.ESCROW_ADDR, abi: escrowAbi } as const;

reader.watchContractEvent({
  ...escrow,
  eventName: "JobPosted",
  onLogs: (logs) => logs.forEach(handleJob)
});

async function handleJob(log) {
  const { id, payment, inputHash } = log.args;

  // 1. Try to claim. First wallet to land their tx wins; the rest revert.
  try {
    await wallet.writeContract({
      ...escrow, functionName: "claimJob", args: [id, BOND]
    });
  } catch { return; }   // someone else claimed first — fine, move on

  // 2. Fetch the prompt off-chain (Tixswarm exposed it at a URL keyed by inputHash).
  const prompt = await fetchPromptByHash(inputHash);

  // 3. Do the work via Claude.
  const title = await runClaude(prompt);

  // 4. Submit result.
  await wallet.writeContract({
    ...escrow,
    functionName: "submitResult",
    args: [id, keccak256(toBytes(title)), toBytes(title)]
  });
}

function runClaude(prompt: string): Promise<string> {
  return new Promise((resolve, reject) => {
    const p = spawn("claude", ["--headless", "--prompt", `Make a title (max 80 chars) for: ${prompt}`]);
    let out = "";
    p.stdout.on("data", (d) => (out += d));
    p.on("close", (c) => (c === 0 ? resolve(out.trim()) : reject()));
  });
}

Deployment uses your existing home-cluster-deploy skill: rsync, launchctl as a user-level service, port 0 (no HTTP — this is purely outbound). Both Mac minis run identical code, only their HERMES_KEY differs.

Why pull the prompt off-chain instead of putting it in the event?

Calldata costs gas. A long ticket description could be many KB; putting it on-chain would cost real money on mainnet. Instead, Tixswarm posts only the hash on-chain (proves the prompt is canonical), and exposes the cleartext at a URL like https://api.tixswarm.com/job-prompt/{inputHash}. Hermes fetches the prompt over HTTPS once it has the hash. This is the standard pattern (it's how IPFS-pinned NFT metadata works too).

Tixswarm Integration Day 1

Three changes to tixswarm's existing code:

  1. Replace the direct Anthropic API call in the title-generation handler with a postJob transaction. Surface the jobId in the response so the frontend can subscribe to its lifecycle.
  2. Add a prompt-resolution endpoint at GET /job-prompt/{inputHash} that returns the cleartext prompt for Hermes to fetch (rate-limit it; nothing sensitive on day 1).
  3. Listen for JobSubmitted events in a long-lived process (Lambda doesn't fit this — needs an EC2/ECS or a Mac-mini sidecar). When a submission arrives, write the title to the ticket, surface the 60-second auto-release countdown in the admin UI with a "Dispute" button.

Lambda doesn't watch websockets

Tixswarm's API is on Lambda (per your deploy-spa-lambda-supabase pattern). Lambda can post to the chain (a short request) but can't listen for events long-term. Cleanest fix: run the event-listening sidecar on macmini64 alongside Hermes. The sidecar writes incoming job-submission events to the tixswarm Supabase DB, and the Lambda frontend reads from there. This is a small but real piece of infra — call it ~30 minutes inside the day's budget.

Risks & Gotchas

Latency

Each on-chain transaction takes ~2 seconds to confirm on Base. A full job is 4 transactions (post, claim, submit, release) plus a 60-second dispute window. That's ~70 seconds minimum per title. Tixswarm's user-facing UX needs the title now, not in 70 seconds.

Day-1 mitigation: optimistic UI. Render the title in the form as soon as the JobSubmitted event fires (step 9 above, ~15s in). Show a small badge: "settling on-chain… 45s remaining." Settlement happens in the background. If the buyer disputes, the title flips back to a placeholder; otherwise it's already correct.

Gas cost vs LLM cost

On Sepolia this is moot (everything's free). On mainnet, each transaction costs ~$0.02 in gas at typical Base prices:

Per jobSepoliaBase mainnetAnthropic API

Total to buyer$0$0.08$0.001

Mainnet gas dwarfs the LLM cost by 80×. This is the most important post-MVP problem to solve. Solutions: batch many jobs into one transaction; use state channels; or shift to "off-chain coordination, on-chain checkpoint" where 1000 jobs settle in a single batch tx. None block the day-1 demo; all will be Week 3+ once the flow is real.

Worker availability

If both Mac minis are down, Tixswarm hangs forever waiting for a job to be claimed. Day-1 mitigation: a 30-second claim timeout in the buyer code that falls back to a direct Anthropic API call. Slightly more work in the integration but eliminates the most user-visible failure mode.

Title quality

Day 1 has no automated quality check. Hermes could submit "asdf" and earn ag3nt unless the admin clicks Dispute. This is fine for a demo (you control both sides) but obviously not viable for an open marketplace. The Week 2+ reputation stack is the answer; for now, just trust your own daemon.

Chain reorgs

L2s can reorganize blocks within a few seconds of submission. Hermes should wait for ~3 confirmations before treating a job as canonically claimed. Failure mode is benign: a duplicate-claim revert costs gas, not money.

Glossary

address
20-byte public identifier of an EVM account, displayed as a hex string starting with 0x. Derived from a public key (for EOAs) or from a contract's creation parameters (for contracts).
Base
An Ethereum L2 (layer 2) network built on the OP Stack and operated by Coinbase. Same EVM as Ethereum, ~100–1000× cheaper gas. "Base mainnet" is the production network; "Base Sepolia" is its testnet.
bytecode
The compiled EVM instructions of a smart contract. Solidity compiles down to bytecode that the EVM executes.
chain ID
Numeric identifier that distinguishes one EVM network from another. Ethereum = 1; Base mainnet = 8453; Base Sepolia = 84532. Prevents transactions signed for one network from being replayed on another.
commitment
A short hash of something larger that proves you knew the larger thing at a given time, without revealing it. Used here to log the prompt hash on-chain cheaply while keeping the cleartext off-chain.
contract / smart contract
A program deployed to the EVM. Has its own address, state, and a set of functions callable by users or other contracts.
EOA (Externally Owned Account)
An EVM account controlled directly by a private key, as opposed to a contract account. The simplest kind of wallet; what we use for the MVP.
ERC-20
Standard token interface on Ethereum. Defines balanceOf, transfer, approve, etc. Almost every fungible token on EVM chains is ERC-20. Our ag3nt is ERC-20.
ERC-4337
Account-abstraction standard. Lets a smart contract serve as a wallet, with custom signature logic, paymasters (third parties paying gas), session keys, and other features useful for autonomous agents. Future work.
EAS (Ethereum Attestation Service)
Standard infrastructure for on-chain attestations: "address A signs claim C about subject S at time T." Used here (in the future) as the substrate for vouches.
event
A log entry emitted by a smart contract. Cheaper than writing storage. Off-chain code subscribes to events via RPC to react to on-chain state changes.
EVM
Ethereum Virtual Machine. The runtime that executes smart-contract bytecode. Used identically on Ethereum, Base, Arbitrum, Optimism, and many other chains.
faucet
A free dispenser of testnet ETH. Required because every transaction (even on testnet) needs gas; faucets give you the gas tokens for free so you can test.
gas
The unit of computation cost on the EVM. Every operation has a gas price; the transaction's total cost = gas used × gas price, paid in ETH (or the chain's native token). "Postage on the envelope" — required to do anything on-chain.
hard cap
A maximum supply limit enforced inside a token contract. Our ag3nt has a hard cap of 21,000,000.
hash
A short fixed-length fingerprint of arbitrary data. We use keccak256 (the EVM's standard hash). Two different inputs almost certainly produce different hashes.
immutable
A Solidity keyword meaning "set once in the constructor, never changed." Also a property of contracts themselves when they have no upgrade mechanism — the bytecode is permanent.
L1 / L2
Layer 1 = the base blockchain (Ethereum). Layer 2 = a chain built on top, inheriting L1's security while running cheaper and faster. Base, Arbitrum, Optimism are L2s.
mainnet / testnet
Mainnet = the production network with real economic value. Testnet = a parallel network with the same software but no real value, used for development. Sepolia is the current main Ethereum testnet; Base Sepolia is its Base counterpart.
mint
To create new tokens. Only functions inside the token contract can mint, and only under whatever rules the contract enforces.
nonce
A per-account transaction counter. Every transaction includes the sender's current nonce; this prevents replay attacks and enforces ordering.
optimistic settlement
A pattern where the happy path requires no action (default = accept), and the only intervention is a dispute. Used in our escrow's 60-second auto-release window.
private key / public key
Asymmetric crypto pair. The private key signs transactions; the public key (and its derived address) is shared openly. Whoever holds the private key controls the account, period.
revert
An EVM transaction failure. Reverts undo all state changes from the transaction but still consume gas. We use require(...) in Solidity to revert on bad inputs.
RPC
Remote procedure call — the HTTP/WebSocket interface to a chain node. Tools like viem and ethers wrap RPC calls. Examples: https://sepolia.base.org, https://mainnet.base.org.
Solidity
The dominant high-level language for writing EVM smart contracts. Compiles to EVM bytecode.
token / native token
Native token = the chain's built-in currency, used for gas (ETH on Ethereum and L2s). Other tokens are contracts deployed on top — ERC-20 tokens like ag3nt are not native, they live inside their own contract.
transaction
A signed message from an EOA that calls a function or transfers value. Costs gas, mined into a block, then visible publicly on the chain.
viem / ethers
JavaScript/TypeScript libraries for interacting with EVM chains from off-chain code. We use viem in Hermes.
wallet
Any tool — software or hardware — that holds a private key and signs transactions on its owner's behalf. Could be MetaMask, a hardware device, or a key in a .env.
