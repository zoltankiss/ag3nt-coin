SPEC audit

Old SPEC.md (TS prototype) vs. current Cosmos chain implementation.
Architecture: changed Primitives: preserved PageRank: implemented

1Summary

The old SPEC.md described a TS prototype with a custom Bun-served HTTP API, JSON block files on disk, in-band signatures, one tx per block, and raw Ed25519 pubkeys as addresses. The current implementation is a full Cosmos SDK chain with CometBFT consensus, protobuf-wrapped txs, Bech32 addresses, and standard gRPC + REST surfaces.

The business logic was preserved: accounts, vouches (with upsert semantics), one-shot faucet, transfers, and PageRank-driven reputation. The architecture changed completely: wire format, address encoding, replay protection, and API protocol are all Cosmos-standard now. Both that and the preservation are deliberate — moving to Cosmos was the decision that kicked off this divergence; that's why we wrote this audit.

This document records the divergence so the rewritten SPEC.html can confidently throw away the prototype-era contract.

3Flagged issues

The high-value findings, in priority order:

1 · Vestigial Account.nonce
The TS prototype required a per-message nonce for replay protection. The Cosmos chain inherits AuthInfo.sequence from x/auth instead, so Account.nonce is stored but never read, written, or validated by any handler. Either delete the field or repurpose it — leaving it is a footgun for the next person who tries to use it. Tracked in SPEC open questions.
2 · ADD discovery (/.well-known/add.json) missing
The TS prototype served this. The Cosmos node has no custom HTTP mux for it. Until we either bolt on a sidecar or extend the node, the chain is not ADD-discoverable. Bundled with the Ed25519↔Bech32 adapter for MVP-2.
3 · PageRank reputation
chain/x/agntcoin/keeper/pagerank.go implements anchor-rooted weighted power-iteration PageRank over the vouch graph (an earlier scaffold returned a uniform 1/N placeholder). Reputation queries now carry real signal.
4 · Wire-format divergence
No client written against the old SPEC's JSON-with-sig tx format can talk to this chain. All clients must build protobuf sdk.Tx messages and submit via /cosmos/tx/v1beta1/txs (or use the CLI / SDKs). The old SPEC's wire format is fully obsolete.
5 · Register idempotency flipped
Old SPEC: re-registering is a silent no-op. New code: re-registering returns ErrInvalidRequest. Harmless either way, but client code that relied on no-op semantics will surface as an error now.
Things that did survive the port cleanly
Account fields (balance, registered, faucet_claimed), vouch upsert semantics by (from, to), faucet amount (10,000 microcoin), max vouch weight (100), one-shot faucet semantics, the agnt address prefix, and the PageRank contract (input shape + output shape, even though the body is a stub).

7Block batching

The old SPEC said "exactly one tx per block in MVP-1 — no mempool yet." The Cosmos chain uses the standard CometBFT mempool and batches as many txs per block as fit.

Why this changed
Cosmos / CometBFT are not configurable to one-tx-per-block at the consensus layer. Forcing the 1:1 ratio would mean writing a custom mempool, which is a worse investment than just accepting batching. The "one tx per block" line in the old SPEC was about prototype simplicity, not a real invariant — it can go.

15Nonce vestigial

The wart
Account.nonce exists in the proto, is set to 0 at registration, and is never touched again. No handler reads or increments it. Cosmos uses AuthInfo.signer_infos[].sequence (managed by x/auth) for replay protection.

Two reasonable resolutions:

  1. Delete it — clean, but a proto field deletion is a breaking change. Since we're pre-launch, this is cheap right now.
  2. Repurpose it — e.g. track per-account tx count for the agent-economy primitives (could feed into demurrage / activity scoring in MVP-2).

Decision deferred to MVP-2 design.

16Faucet authorization

Old SPEC

"Signed by the agent themselves (not the validator). The validator mints into existence on apply."

Implementation

The agent signs the tx. Cosmos verifies the signature in antehandler before the message reaches the handler. The handler then mints by incrementing Account.balance. Same mechanism in spirit, different machinery.

Status: partial — semantic match, structural difference.

18Recommendation

Done
The old SPEC.md has been replaced by SPEC.html, which describes the Cosmos reality. This audit is the historical bridge — keep it for context, don't try to drag SPEC.md back to consistency. The TS prototype still exists in ts-reference/ and can be the spec-as-implemented for that directory in isolation.

Open items rolling into MVP-2 design:

  • Decide what to do with Account.nonce (delete or repurpose).
  • Ed25519↔Bech32 adapter for ADD-native callers.
  • /.well-known/add.json discovery (sidecar or mux extension).
  • Implement anchor-rooted weighted PageRank (done — see keeper/pagerank.go).
