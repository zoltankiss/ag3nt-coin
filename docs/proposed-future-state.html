Proposed future state

What MVP-2, MVP-3, and v1 should look like. Provisional, opinion-shaped, designed to be argued with.
2 declared jibes 3 new modules demurrage rewrites incentives

The arc

MVP-1 proved the primitives — accounts, vouches, faucet, transfers — work on a Cosmos chain with a single validator. The next three milestones do three different things:

  • MVP-2 closes the credibility loop. ADD-native callers can talk to the chain. Misbehaving agents can be tried and slashed. Coins decay if you sit on them. This is when the chain becomes opinionated about agent economics.
  • MVP-3 gives coins a peg to something real (FLOPs of useful compute) and a way to earn them by doing work, not just being vouched-for. This is when the chain becomes useful outside the ADD ecosystem.
  • v1 opens the doors. Multi-validator PoS, public chain, IBC. This is when it becomes real.

Trust, cost, and ADD-native access

Next milestone

ADD-native adapter new

ADD agents identify with raw Ed25519 base64 pubkeys; the chain identifies with Bech32. A sidecar HTTP service translates the two, so ADD-native callers don't have to learn Cosmos address encoding.

Design choices

  • Sidecar, not in-process. The Cosmos node stays pristine — adapter outages don't take down consensus, and the adapter can be upgraded independently. It can run on macmini64 next to the validator.
  • Stateless. No DB. The adapter does pure translation; all state lives on the chain.
  • Signing options. Either (a) the adapter holds a sponsor key and signs on the agent's behalf using a meta-tx pattern, or (b) the adapter just proxies — agents sign Cosmos txs locally but use the adapter for ADD-shaped request/response. (a) is more agent-friendly; (b) is more honest about who's signing. Start with (b); promote to (a) only if friction proves it's needed.

Trials module (x/trials) new module

The credibility loop. Any agent can file a complaint against another. A jury of randomly-but-reputably selected peers reviews evidence and votes; guilty verdicts slash the defendant's balance into a pool that funds future faucet drips.

Jury selection JIBE #2

Pick k jurors from the population of registered agents, weighted by reputation score, without replacement, with a single pass over the population.

Why this is a jibe
Weighted reservoir sampling without replacement (the A-Res / A-ExpJ algorithms) is classic interview-worthy territory: clever use of order statistics on exponentials, streaming-friendly, one pass. The boring version (compute CDF, binary search, repeat k times with rejection) works but is twice the code and miles less elegant. The human implements this; the agent wires the call site.

Contract

func SelectJury(
    rng *rand.Rand,         // seeded deterministically from block hash + trial id
    candidates []JuryCandidate, // {address, weight float64 (reputation score)}
    k int,                  // jury size, e.g. 7
) []string                  // k addresses
  • Determinism is mandatory. Every validator must select the same jury from the same inputs. Seed the RNG from blockhash || trial_id.
  • Population: all registered agents with reputation > 0, excluding plaintiff and defendant.
  • Edge case: if |candidates| < k, fall back to "select all of them" (don't error — small early populations are common).
  • Hint for the jiber: Algorithm A-Res (Efraimidis–Spirakis) — for each candidate i, compute key u_i^(1/w_i), keep the top-k by key. One pass, O(n log k) with a min-heap of size k.

Demurrage incentive flip

Coins decay at a fixed rate per block of inactivity. Decayed coins accrue to the network's faucet pool, which funds new agents' first drips and (eventually) jury rewards.

Design choices to settle

  • Decay trigger: per-block continuous decay (math everywhere, gas cost on every interaction) vs. lazy decay (compute on read/write; store last_touched_height per account). Lazy is right — it's cheaper and equivalent.
  • "Activity" definition: any tx from the account resets the clock? Or only certain tx types (transfer counts, vouch doesn't)? Probably "any tx."
  • Floor: never decay below 100 microcoin so dormant accounts don't drop to zero (they'd become un-faucetable since they've already claimed).
  • Decay pool: tracked as a module-level counter; MsgFaucet first tries to source from it before minting fresh supply.

Decide what to do with Account.nonce

From the audit: the field is vestigial. Two paths:

Option A · Delete
Cleanest. Removes the trap for the next reader. Costs: proto field deletion is a breaking change. Pre-launch, this is essentially free.
Option B · Repurpose as activity_count
Increment on every tx from the account. Feeds into demurrage's "last active" notion and could weight jury selection toward agents who actually use the chain. Cheap to do and turns dead code into useful signal.

Recommend B. Repurposing the field at the same time we're adding demurrage avoids a separate breaking change later, and it gives us an obvious fairness signal for jury selection.

MVP-3

Compute marketplace

After MVP-2 ships

Proof-of-Useful-Work

Agents earn coin by running inference jobs for the network. The chain coordinates job posting, assignment, result submission, and verification. The "work" is real (inference FLOPs), not arbitrary hashing — hence "useful."

Open question · is k-of-n enough?
Redundant execution catches lazy/lying workers as long as the majority is honest. It doesn't catch a coordinated cartel. Two mitigations: (a) random worker selection weighted by reputation (reputation is now meaningful — trial losers can't take jobs), (b) periodically inject canary jobs with known answers. Both can be added incrementally.

FLOPs peg

1 ag3nt-coin = N FLOPs of useful inference (target). This anchors the coin to a real, measurable resource — the way wheat anchored medieval currencies — without requiring an oracle to USD.

  • How emission works: when a job completes, the chain mints the bid amount and pays workers. The "mint" is governed so the average coin/FLOPs ratio drifts toward the peg.
  • Why FLOPs (not USD): agents care about being able to do inference, not about dollar prices. FLOPs is the native cost basis of an agent's activity. Pegging to it makes coin balances directly translate to "how much thinking can I afford."
  • Tradeoff: FLOPs measurement is approximate (model+hardware dependent). Acceptable noise for MVP-3; could be refined with hardware attestation or canary benchmarks later.

Multi-validator PoS testnet

Drop the single-validator simplification. Wire up x/staking properly with agntcoin (or stake denom) as the bonded asset. Run a public testnet with ~10 geographically-distributed validators.

  • Genesis distribution: some pre-mine for bootstrap validators, majority emitted via PoUW / faucet over time. Keep founder allocation small to preserve "by agents, for agents" credibility.
  • Slashing: standard Cosmos slashing for double-sign and downtime. Trial-based slashing (MVP-2) is separate from validator slashing.
  • Governance: standard Cosmos governance module. Anything tuned (demurrage rate, faucet amount, jury size, FLOPs peg) becomes a governance parameter.

Public mainnet

  • IBC: enable the IBC module so agntcoin can hop to other Cosmos chains. Most useful for paying ADD agents that earn on agntcoin but spend on chains with deeper liquidity.
  • Bridges: probably not in v1. IBC covers Cosmos; non-Cosmos bridges are a security liability we don't need yet.
  • Hosted indexer: not part of the chain, but worth scaffolding — most consumer-facing UX needs a chain indexer (PostgreSQL-backed projection of chain events). Could run on macmini64 alongside the validator until it outgrows it.
