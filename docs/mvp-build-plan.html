0 · Reality check (read this first)

"TixSwarm" is real — it's add-native-ticket-tracker (GitHub add-ticket-tracker; Bun + SAM/Lambda). It was just under a different name. The integration seam is exact and already isolated: 
// src/tickets/title-generator.ts

  // today: calls Claude Haiku in-process
  // later: post a job to JobEscrow, await Hermes' result — same return type
}

We still build a thin Agent A buyer stand-in for day 1 (you approved this) rather than modifying the production ticket tracker immediately. The stand-in posts the exact job generateTitle would ("title this ticket body") through the exact on-chain interface. That keeps the live product untouched while we prove the loop. Swapping in the real consumer later is a one-function change to generateTitle — zero contract or daemon rework. The e2e is genuinely real today: real contracts, real Base Sepolia, real Claude doing the work, real settlement.

Verification-hardness caveat (your own May 24 flag). A job marketplace only works when the buyer can cheaply verify the worker's output without redoing the work. For most LLM tasks, verifying ≈ solving — fatal. Title generation is verification-cheap (the buyer eyeballs / best-of-N judges in a 60s dispute window), which is exactly why it's the first job type. The MVP does not claim to solve verification in general; it picks a task where verification is easy.

4 · Hermes worker daemon Bun + viem

// the loop, in spirit
watch JobPosted →
  if profitable && idle: claim(jobId)   // stakes a small bond
  on JobClaimed(me):
    title = await claude(`-p`, jobBody)  // the actual work
    submit(jobId, hash(title), title)
  // escrow releases payment after 60s window or buyer's early accept

Idempotent and crash-safe: state is re-derived from chain events on boot, so a restart never double-claims or loses a job. One codebase runs on both Mac minis with different keys — they genuinely compete.
