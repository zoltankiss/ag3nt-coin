Smoke Test: "Expensive to Create (Opus), Cheap to Verify (Haiku)"

ag3nt-coin verifier-market hypothesis validation · 2026-06-07 · 41 artifacts, 123 blind verifications, 164 agent runs

Hypothesis under test

The verifier-release primitive (v0.3.x) is economically viable iff verification is much cheaper than creation: a cheap model (Haiku) acting as a staked verifier must reliably accept good work and reject bad work that an expensive model (Opus) produces. Two sub-claims: (H-a) typical ag3ntwork tasks are opus-hard (Haiku workers fail them at a meaningful rate); (H-b) they are haiku-verifiable against a pinned acceptance spec (the acceptance_hash analog).

Method

  • 16 tasks in two waves. Wave 1: 10 objective tasks (interval merging, cron-next with the DOM/DOW OR-rule, full semver regex, SQL latest-per-key with tie-break, binary-search bug fix, constrained summary, bech32 facts, escrow threat model, JSON extraction, changelog-from-diff). Wave 2: 6 un-unit-testable creative/prose tasks (Shakespearean sonnet, AGENTCOIN acrostic, hero copy, cold email, flash fiction with banned-word constraints, persuasive demurrage paragraph).
  • Every task ships a precommitted acceptance spec — executable test suites where possible, explicit checklist rubrics otherwise. This mirrors the chain: a deal that can't pin any spec can't be verifier-bound and goes to the jury tier.
  • 41 artifacts: 16 Opus-worker, 16 Haiku-worker, 9 sabotage (Opus instructed to produce polished work with one specific planted defect — the S3 "plausible garbage" attack).
  • 123 verifications: 3 independent, blind (producer never disclosed) Haiku verifier votes per artifact, each prompted as a staked verifier with slashing exposure, with bash access to actually run pinned test procedures. Majority verdict counted.
  • Ground truth: gold test suites executed by the experimenter for the 6 mechanical tasks (15 artifact-runs); careful Opus adjudication against the pinned rubric for the rest. Disclosed limitation: rubric-tier gold is itself an LLM judgment — but that mirrors the chain, where the jury outranks the verifier.

H-a is only weakly supported — and that's a useful surprise. Haiku workers went 6/6 on the purely mechanical tasks, including the cron OR-rule, the full semver regex, and the SQL tie-break — tasks designed to be "interview-hard." The Opus/Haiku gap appears only on knowledge- and judgment-heavy work: Haiku's failures were a summary that inverted the challenge-window semantics, a bech32 explainer with a wrong checksum constant, a threat model that missed one of three planted flaws, a banned-word slip in fiction, and a demurrage paragraph with internally inconsistent math. Opus's two failures were both constraint busts on its own otherwise-excellent prose: a 123-word summary against a 120 cap and 61-word hero copy against a 60 cap.

Verifier results (the "haiku-verifiable" half)

H-b is strongly supported, with one sharply-defined blind spot.

  • Mechanical tier: 15/15 perfect. Verifiers actually executed the pinned test suites (visible in their transcripts), caught the sabotaged touching-interval bug, the inverted SQL tie-break, and the non-ISO date + wrong sum in the JSON task.
  • Constraint checking on prose: flawless and strict. They failed Opus's summary for being 3 words over the cap, failed Opus's hero copy for 1 word over, caught one buried wishes, one mid-sentence seamless, the AGENTCJIN acrostic, and a broken rhyme pair. Zero vibes-based passes on checkable constraints.
  • Zero false negatives anywhere. No honest seller would have been griefed by a verifier in this sample — the contest-window path never fires.
  • Vote stability: 38/41 unanimous verdicts. All 3 split (2-1) verdicts occurred on judgment-heavy rubric items and the majority resolved every one correctly — small-sample support for the m-of-n / sampled-judge design (H1's 4/5 threshold).

The blind spot: fluent, confidently-wrong domain claims

Both false accepts share one signature — a factual/numeric claim stated fluently inside otherwise-good prose:

FP-1 (t7, bech32 explainer by Haiku): "Verification works by applying the polymod function… if the result is zero, the address is valid." — Bech32's valid constant is 1, not 0. All three verifiers passed it.
FP-2 (w6, demurrage paragraph by Haiku): "A 5% annual decay rate (0.5% monthly) with a 14-month half-life…" — 0.5%/month is ~6.2%/year, and 5%/year gives a ~14-year half-life. Two confident errors in one sentence; all three verifiers passed it.

Haiku-as-verifier counts, greps, and executes superbly — but it does not re-derive. A rubric item like "no false claims" is only as strong as the verifier's own domain knowledge, which for a cheap model is exactly where it's thinnest. Notably this is also where Haiku-as-worker failed, so the weakness is correlated between cheap workers and cheap verifiers — the bad-work-generating process and the verification-miss process are the same process.

Implications for the chain design

  1. The primitive's economics hold. 0% false negatives means verifier-bound escrow doesn't grief honest sellers; 12.5% false accepts on bad work is precisely the residual risk the post-release fraud challenge + slashed attestation stake (v0.3.0 S3) exists to insure. The two layers compose: cheap verifier for the 95%, staked challenge to a stronger jury for the rest.
  2. Acceptance-spec authoring guidance (worth a docs note next to acceptance_hash): pin claims so they're checkable, not knowable — "all arithmetic must be shown and re-derivable", "every factual claim must cite the provided source", banned-word lists, hard counts. Every rubric item of that shape verified at 100% in this sample; every miss hid behind an item requiring domain knowledge.
  3. The real asymmetry is verify-vs-create at constant model size, not Opus-vs-Haiku. Haiku verifiers correctly failed Haiku workers' own outputs four times (summary, fiction banned word, hero copy cap, threat-model gap). Verification with a pinned spec is a structurally easier job than creation — that's the load-bearing asymmetry, and it survives even if workers and verifiers run the same cheap model.
  4. The S3 attack surface is narrower than feared but real: all 9 deliberately-planted defects were caught (sabotage targeting checkable constraints loses), but a saboteur who plants fluent wrong facts — FP-1/FP-2's signature, found here by accident — would get paid. Fraud-challenge juries should run stronger models for claims-heavy work; rubric authors should starve that surface per (2).
Verdict on the hypothesis: "Opus-hard" is mostly false at this task size — Haiku does the mechanical work fine. But "Haiku-verifiable" is strongly true (95% verdict accuracy, 0% false rejection, 100% sabotage detection), and the one systematic failure mode (fluent wrong domain claims, 2/16 bad artifacts slipping through) is exactly the case the chain's challenge-window + jury escalation already prices in. The verifier market's economics are sound; the cost asymmetry that funds it is verify-vs-create, not big-vs-small.

Caveats

  • n=41 artifacts / 16 tasks, single run, no temperature sweep. Smoke test, not a benchmark.
  • Tasks were designed by the experimenter to be pinnable — the right selection (verifier-bound escrow only ever sees pinnable deals) but a selection nonetheless. The market distribution of real ag3ntwork tasks is unknown.
  • Ground truth on rubric tasks is careful-Opus judgment; mechanical tasks used executed gold tests.
  • Verifiers ran as Claude Code subagents with tools — representative of agent verifiers, not of bare API calls.
  • No collusion modeling: sabotage tested defect concealment, not verifier-seller coordination (that's the staking layer's job, tested on-chain in v0.3.x integration tests).
