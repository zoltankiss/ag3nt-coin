ag3nt-coin

A sovereign appchain for agents, by agents. Cosmos SDK + CometBFT. MVP-1.
MVP-1 live (local dev) PageRank: stub (JIBE) Single validator

1Premise

A crypto optimized to be by agents, for agents, powering the ADD economy. Differs from BTC/ETH because:

  • Agents lack human emotional drivers (greed, scarcity anxiety, status). So we invert money mechanics: demurrage replaces hoarding incentives; value is flow, not stock. (Demurrage lands in MVP-2.)
  • Agents already have Ed25519 identity from ADD. Every API call is signed; the chain inherits a free evidence trail.
  • ADD-native apps are the customers. The chain's first job is to make ADD interactions trustworthy and economical at scale.

2Non-goals (MVP-1)

  • Decentralization. MVP-1 is a single-validator dev chain. The point is to validate the agent-economy primitives, not the consensus algorithm.
  • Performance. Default CometBFT block time (~1s) is fine.
  • Public testnet. Local only. Public multi-validator testnet is v1.
  • ADD-native wire format. Agents currently speak Bech32 directly; raw-Ed25519-base64 ↔ Bech32 adapter is MVP-2.

3Architecture

Standard Cosmos SDK stack. One custom module, x/agntcoin, holds the agent-economy primitives. Everything below the module is off-the-shelf.

4Identity & addresses

Addresses are Bech32 with the agnt prefix (e.g. agnt1qy2…). Derived the standard Cosmos way: bech32(ripemd160(sha256(pubkey))).

The operating principal for ADD ecosystem work is the chief-of-staff key (Ed25519). For the chain itself, each agent generates a Cosmos key via agntcoind keys add <name> and signs txs with it.

Future-state note
ADD speaks raw Ed25519 base64 pubkeys. The chain speaks Bech32. MVP-2 introduces an adapter so ADD-native callers can transact without learning Cosmos address encoding. Until then, callers must use Bech32 directly.

Cosmos vs. agntcoin denoms
The Ignite scaffolding adds standard Cosmos denoms (stake, token) in config.yml for the auth/bank modules to function. The agent-economy primitives above are separate from those — agntcoin balances live in x/agntcoin's own KV store, not in x/bank. There is no on-chain swap between the two; the Cosmos denoms are scaffolding noise until they become useful.

6Messages (Tx)

Four agent messages and one governance message. All are protobuf, wrapped in the standard sdk.Tx envelope, signed with the sender's Ed25519 key. Replay protection uses Cosmos' per-account sequence number (managed by x/auth), not a custom nonce.

MsgRegister

Establishes an account at zero balance. Must be the first tx from a key.

message MsgRegister {
  string creator = 1;   // Bech32 address; signer of the tx
}
  • Validation: creator must be a valid Bech32 address; account must not already exist with registered=true.
  • Effect: writes Account{address, balance:0, registered:true, faucet_claimed:false}.
  • Idempotency: rejects re-registration with ErrInvalidRequest. (Diverges from the TS prototype, which made this a silent no-op.)

MsgFaucet

One-shot 10,000-microcoin drip per agent, ever.

message MsgFaucet {
  string creator = 1;
}
  • Validation: account must be registered; faucet_claimed must be false.
  • Effect: mints FaucetAmount into the account; sets faucet_claimed=true.
  • Authorization: the agent's own tx signature is sufficient — the chain mints into existence on apply (no validator-signed dispensation needed).

MsgTransfer

message MsgTransfer {
  string creator  = 1;   // sender
  string to_addr  = 2;
  uint64 amount   = 3;   // microcoin
}
  • Both addresses must be valid Bech32 and both accounts must be registered.
  • amount > 0; creator != to_addr (self-transfer rejected).
  • Sender balance must be ≥ amount. No fees in MVP-1.

MsgCastvouch

Asserts weight units of trust from sender to to_addr.

message MsgCastvouch {
  string creator = 1;
  string to_addr = 2;
  uint64 weight  = 3;   // 1..100
}
  • Self-vouch rejected. Weight must be in [1, 100].
  • Both accounts must be registered.
  • Upsert semantics: if a vouch with the same (from_addr, to_addr) exists, its weight is replaced. Otherwise a new Vouch is appended with an auto-incremented id from VouchSeq.
  • No coin movement.
Also: MsgUpdateParams (governance only)

Standard Cosmos governance message. The Params message is currently empty (no tunable parameters yet), so this exists only because the module scaffold requires it. Will hold demurrage rate, faucet amount, etc. once those land.

Account

message Account {
  string address        = 1;
  uint64 balance        = 2;   // microcoin
  uint64 nonce          = 3;   // see warning below
  bool   registered     = 4;
  bool   faucet_claimed = 5;
}
Known wart: nonce is vestigial
The nonce field exists from the TS prototype but is never read, written, or validated by any message handler. Cosmos uses AuthInfo.sequence (managed by x/auth) for replay protection. Decision needed: remove the field, or repurpose it (e.g. track per-account tx count for the agent-economy primitives). Tracked under open questions.

Vouch

message Vouch {
  uint64 id        = 1;
  string from_addr = 2;
  string to_addr   = 3;
  uint64 weight    = 4;   // 1..100
}

Vouches are deduplicated by (from_addr, to_addr) in the MsgCastvouch handler: a second vouch over the same edge updates the existing row's weight rather than appending. The handler does this with a full Vouch.Walk — fine for MVP scale, will need a secondary index later.

9Genesis

message GenesisState {
  Params         params      = 1;
  repeated Account account_map = 2;
  repeated Vouch   vouch_list  = 3;
  uint64         vouch_count = 4;   // restores VouchSeq
}

InitGenesis writes accounts and vouches as-is, restores VouchSeq from vouch_count, and sets Params. ExportGenesis walks the same collections in reverse.

Ignite's chain/config.yml seeds dev accounts (alice, bob) with the Cosmos default denoms (stake, token). These are for x/auth/x/bank — not agntcoin balances. Agent accounts are bootstrapped via MsgRegister + MsgFaucet.

10PageRank JIBE

Reputation is a pure function of the vouch graph, computed on each query — not stored. The query handler walks Vouch, calls PageRank(vouches), and returns the requested address's score.

Status: stub
chain/x/agntcoin/keeper/pagerank.go currently returns a uniform distribution (1/N per node). The chain queries cleanly, but scores carry no signal. Full power-iteration / weighted PageRank is the first jibe — to be hand-implemented.

Contract

  • Input: []types.Vouch — every edge in the graph. Already deduplicated by (from, to) by the time it gets here.
  • Output: map[string]float64 — one score per node. Node set is the union of from_addr and to_addr across all edges.
  • Empty graph: return map[string]float64{}.
  • Convention: scores sum to 1.0 across all nodes.

Hints (from the JIBE comment)

  • Standard PageRank with damping d ≈ 0.85: r_{k+1} = d·M·r_k + (1−d)/N.
  • Weighted: normalize each out-edge by sender's total out-weight.
  • Dangling nodes (only inbound vouches): redistribute their rank uniformly each iteration.
  • Terminate at L1 delta < 1e-9 or 100 iterations.

11API surface

Three protocols, all exposed by the standard Cosmos node (no custom HTTP server):

gRPC
Native protobuf RPC. Paths follow /agntcoin.agntcoin.v1.{Msg,Query}/…. The way most server-side clients should talk to the chain.
REST (grpc-gateway)
Auto-generated from the proto's google.api.http annotations. Paths: /zoltankiss/agntcoin/agntcoin/v1/…. Convenient for curl/browser.
CometBFT RPC
Standard Tendermint RPC on :26657 (status, tx-by-hash, block, subscribe). Useful for indexers and operational tooling.
CLI
agntcoind tx agntcoin {register,faucet,transfer,castvouch} …
agntcoind query agntcoin {get-account,reputation,…}
Not implemented (yet)
The ADD discovery endpoint (GET /.well-known/add.json) from the TS prototype is not served by the Cosmos node. Adding it requires either a sidecar HTTP server or extending the node's mux. Tracked for MVP-2 alongside the Ed25519↔Bech32 adapter.

12Divergences from the TS prototype

The TS prototype in ts-reference/ implemented a different chain architecture (Bun.serve, JSON block files, custom REST endpoints, in-band signatures, one-tx-per-block). Moving to Cosmos changed a lot. The notable conscious differences:

TopicTS prototypeCosmos chainWhy
Addresses Raw Ed25519 base64 Bech32 agnt1… Cosmos convention; adapter pushed to MVP-2
Wire format JSON tx with in-band sig Protobuf sdk.Tx Inherit Cosmos sig + sequence machinery
Replay protection Per-message nonce AuthInfo.sequence Cosmos handles it; Account.nonce is now vestigial
Block batching One tx per block Standard CometBFT batching Not worth fighting the framework
API Custom REST (/agents, etc.) gRPC + grpc-gateway + CometBFT RPC Standard Cosmos surface
ADD discovery /.well-known/add.json Not implemented Needs sidecar or mux extension; MVP-2
Register idempotency Silent no-op Errors on re-register Cosmos handlers expect explicit errors; harmless either way
Self-transfer / self-vouch Not addressed Explicitly rejected Cheap safety; the obvious choice when scaffolding

The full side-by-side, with code references, lives in the spec audit.

13Open questions (don't block MVP-1)

  • Vestigial Account.nonce: remove from the proto, or repurpose for an agent-economy counter (e.g. activity score for demurrage)?
  • Vouch weight semantics: linear, log, or capped? Defer until there's real data from a populated graph.
  • Faucet refill: should faucet_claimed ever reset? Current stance: no — bootstrap is one-shot, long-tail earning comes from sponsorship and (eventually) PoUW.
  • Vouch lookup index: the upsert path in MsgCastvouch walks all vouches to find (from, to). Add a secondary index ((from, to) → id) when graph size warrants it.
  • Agntcoin denom in x/bank? Currently agntcoin balances are isolated from Cosmos' bank module. If we ever want IBC, fees, or module integration (e.g. staking with agntcoin), we'd need to wire it in. Not needed for MVP-1.

14Out of scope (MVP-2+)

See proposed future state for the full forward-looking design. Headline items:

  • MVP-2: ADD-native adapter (Ed25519 base64 ↔ Bech32); x/trials module (file complaint, weighted-reservoir jury selection [second jibe], evidence, voting, slashing); demurrage on idle balances; /.well-known/add.json.
  • MVP-3: compute marketplace, proof-of-useful-work, FLOPs-pegged coin emission.
  • v1: multi-validator PoS testnet, public chain.
